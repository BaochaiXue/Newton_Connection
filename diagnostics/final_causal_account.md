# Final Causal Account

Date: 2026-04-08

## Question

Why does the robot still collapse or table-load early, even after the old
overwrite path is removed?

## Causal account

The failure is now narrow and causal:

1. The old readable baseline hid the problem by imposing joint state.
2. The native-style rewrite removed that hidden kinematic support.
3. Under the current bridge-side solver/actuation route, the robot cannot
   stably hold the intended `PRE` pose against gravity.
4. Because the robot cannot hold `PRE`, the finger reaches the table before the
   intended contact/push schedule.
5. Once early table loading happens, readable finger-to-rope motion is lost.

## Physics explanation

### Why state overwrite destroys contact-generated tracking error

If controller logic directly writes `joint_q` or body pose after the solver
step, then the solver’s contact impulse cannot persist into the next state.

That means:

- contact can be computed
- but its consequence on actual robot motion is erased

So a blocking table cannot truly “push back”.

### Why true actuation exposes articulation sag

In a truthful controller path, motion is generated by:

- target spring/damper terms
- articulated dynamics
- gravity
- contacts

If the target servo is not strong/stable enough to balance gravity at the
desired pose, the robot will sag until some contact or geometric limit absorbs
the load.

That is exactly what the rewrite reveals.

### Why a table can only push back if motion is solver-executed

A rigid table blocks penetration by generating contact constraints / impulses.
Those only matter if the robot’s next state is actually produced by the solver.

If the next state is imposed externally, the table cannot stop the robot in a
physically meaningful way.

### Why rope motion must be delayed until visible finger contact

For a meeting-honest demo, rope motion is only causal if:

- the finger is the actual contactor
- the rope begins moving after that visible contact

If the robot table-loads first, then the visible story becomes:

- robot is resting/slumping
- rope motion, if any, no longer reads as a purposeful finger push

### Why long support/loading events read as collapse

A long support event means the viewer sees the robot *resting* on a support
surface rather than *interacting through* it.

Visually that reads as:

- slump
- bracing
- resting

not:

- deliberate approach
- purposeful blocking
- controlled push

## Evidence from the rewrite

The v2 rewrite removed the main false explanations:

- no joint-state overwrite in the sim loop
- no support box by default
- no visible tool
- same-history render path
- measured EE reference offset instead of the earlier hard-coded mismatch
- reachable seeded phase skeleton instead of an unreachable Cartesian fantasy

And yet the current candidates still show:

- `robot_table_first_contact_phase = pre`
- zero readable finger-to-rope contact
- real nonzero blocking error

This means the remaining blocker is not “bad metrics”, not “render mismatch”,
and not “legacy hidden helper”.

It is:

**the current articulated actuation surface on the present bridge-side solver
route still cannot hold a clean startup pose in the final truthful
architecture.**

## Bottom line

The rewrite disproved the easy explanations and isolated the real one.

The remaining problem is not cosmetic and not a small patch target.
It is architectural:

- the robot side needs a more native articulated pattern than the current
  bridge-side route provides
- while the rope should stay on the deformable side

That is why the next honest step is a split-architecture experiment rather than
further micro-tuning the same route.
